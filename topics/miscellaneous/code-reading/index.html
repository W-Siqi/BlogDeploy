    关于阅读代码

    最近刚进组，接需求前，需要在熟悉项目的代码。在程序员的职业生涯中，阅读代码有时候会占工作很大的一部分，而且很多问题，都是出在没有正确理解代码上。因此觉得有必要总结一下关于阅读代码这门技术的经验之谈。
    （这是一篇很主观的经验之谈，存在本人境界的局限）

    阅读代码的经验之谈

    带着问题和猜测看

    除非你一上来要整个全局的系统性的大重构，很多时候阅读源码的情形是: "你想要了解这系统的某一功能"。很多大工程的代码量级有几十万行或者上百万行之多，但你真正关心的代码可能只有其中的沧海一粟。那么怎么快速找到你真正关心的代码呢？

    所以看代码前我经常会问自己一些问题： - 我想看哪部分的功能的代码实现？他是怎么做的？ - 如果让我实现，我会怎么做？

    带着这种问题，会让看代码有了很明确的目的性。 举个例子，我想了解游戏内UI开发的的流程，他一般是怎么实现的？ 我定义了我想看的方向，然后再问自己: 如果让我写，我会怎么做？ 我猜现在比较通用的都是MVC的模式，大概会有一个viewer的角色，尝试找到他！大概会有一个从model数据显示到viewer的流程，尝试找到他！ 大概也会有一个玩家点击后，请求后台，再到回调中处理结果的流程，尝试找到他!

    其实这里就体验的一个业务经验积累的价值，如果你在某个领域有经验，甚至做过，那么你对相同的业务代码，有着比较准确的预判和猜测，看代码的过程会更加水到渠成。因为实现同一功能，代码有一万写法，但架构和设计模式不是

    总结一下就是，看代码一定是带着目的去看的，而不是打开代码文件从第一行一行行看。

    细节层次的把控

    在看代码的过程中，很容易先钻进一些细节中去。细节可以在你真正动手写的时候再慢慢看，因为如果一开始就陷阱细节，会对整体的理解把握帮助。而且细节你是永远看不完的。
    所以在初次看代码时，得先定义一下细节的边界，遇到越过边界过于细节的东西，就把他当做一个黑箱。

    一些tips

    了解业务本身

    很多业务实现的功能就复杂，上手使用可能都有很长的文档(比如编辑工具类的东西）。这种你得先了玩一下这个业务实现的功能会比较好。

    工欲善其事必先利其器

    好的工具的重要性，我觉得甚至是大于个人水平的。有的基础功能，比如断点，查找代码引用，全局搜索，这些至少得给力。

    事实的检验的唯一标准

    光看代码本身，停留在纸上谈判的理解是不可靠的！
    把工程跑起来，操作一下打打断点看下对应的代码，更理解深刻。
    而且没事改改代码，或者写一些hello-wrold级别的流程跑一下，验证一下你的理解。

    关于笔记

    我习惯看代码的时候旁边记一下东西，因为信息量很大，所以得记; 又因为信息量很大，不必记太细，很多东西脑子有个印象就ok了。

    一般我的笔记里90%都是两类东西：核心流程 和 核心概念(核心class的职责，设计理念...) 流程捋清你帮助你理解他是怎么运作的，捋清概念帮助你理解他的设计。
    总的来说，就是很高层，大局的东西，稍微记一下，毕竟有的代码一天看不完。

    要理解代码的不完美

    在频繁迭代，人员也会流动的项目中，代码有很多冗余的，迷惑的，废弃的设计是很正常的。这得理解，但反应在阅读代码的过程，就是多一点戒心，你不要只看一两个例子，就归纳出这类业务的通用写法，可能你看的例子是躺在人家的重构列表里的。
